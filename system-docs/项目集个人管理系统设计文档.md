# 项目集个人管理系统设计文档

## 1. 设计目标

`projects-personal-manager` 是一个本地项目集管理系统，用于集中记录个人代码项目、项目组、阅读材料和 AI 协作入口。

当前系统基于 OMP 组织 AI 协作协议。系统目标是：

- 索引项目和项目组。
- 保留项目事实源和阅读材料。
- 通过 OMP skills 组织 AI 工作流。
- 用极短显式词元控制 AI 当前应执行的流程。
- 尽量减少常驻上下文和无关 workflow 读取。

系统不承担以下职责：

- 不统一管理各项目的构建、测试、发布或依赖。
- 不接管各项目自身 Git 仓库。
- 不规定项目内部目录结构。
- 不把 AI 生成摘要或讨论材料作为项目事实源。

## 2. 当前目录结构

当前核心结构如下：

```text
projects-personal-manager/
├── AGENTS.md
├── README.md
├── package.json
├── tsconfig.json
├── registry.yaml
├── .omp/
│   └── skills/
│       ├── discuss/
│       ├── design/
│       ├── implement/
│       ├── fix/
│       └── review/
├── projects/
├── programs/
├── reading-resources/
│   ├── projects/
│   └── 未分类/
└── system-docs/
```

说明：

- `AGENTS.md` 是 AI 接入本仓库时的常驻协议入口。
- `.omp/skills/` 存放 OMP skills，每个 skill 一个目录。
- `projects/` 存放接入项目，每个项目至少提供 `.project.yaml`。
- `programs/` 存放项目组，每个项目组使用 `_program.yaml` 描述。
- `registry.yaml` 是项目集索引，当前保留为本地生成或维护结果。
- `reading-resources/` 存放适合人类阅读的材料和 AI 讨论/设计草稿。
- `system-docs/` 存放本系统自身的正式设计文档。

## 3. OMP 接入协议

### 3.1 `AGENTS.md`

`AGENTS.md` 只保留极短常驻协议，负责说明：

- 当前环境。
- skill routing 语法。
- active skill 的生命周期。
- 不通过语义相似度激活 skill。
- 激活前只读取 skill metadata，不读取 workflow body。

当前路由语法：

```text
[X]  activate skill matched by X
[X!] reload skill matched by X, discard workflow drift, continue current task
[E]  deactivate current skill, continue normally
```

其中 `X` 不是 skill 名，而是 route token。AI 应将 `X` 与 OMP skill metadata 中声明的 route token 匹配。

### 3.2 Active Skill

active skill 是当前任务的工作流状态。

规则：

- 只能通过显式 routing syntax 激活 skill。
- active skill 会持续到退出、切换或重载。
- active skill 决定用户后续消息的解释方式。
- 普通自然语言不能触发 skill 切换。
- `[X!]` 用于纠偏：重新进入对应 skill，丢弃偏离 workflow 的临时状态。
- `[E]` 用于退出：退出当前 skill 后，同一条消息剩余内容按普通请求处理。

### 3.3 Skill Metadata

每个 skill 的 route token 由 `SKILL.md` frontmatter 中的 `description` 声明。

当前约定格式：

```yaml
---
name: skill-name
description: "route tokens: [X]"
---
```

当前实现不在 `AGENTS.md` 中维护具体 token 到 skill 的映射。这样新增或修改 skill 时，只需要修改对应 `SKILL.md`。

## 4. 当前 OMP Skills

当前 skills 只在本设计中登记入口和职责边界；具体 workflow 以各自 `SKILL.md` 为准。

| Token | Skill | 路径 | 简要职责 |
| --- | --- | --- | --- |
| `[D]` | `discuss` | `.omp/skills/discuss/SKILL.md` | 澄清问题、探索方向和关键取舍 |
| `[P]` | `design` | `.omp/skills/design/SKILL.md` | 基于已确认方向整理设计方案 |
| `[I]` | `implement` | `.omp/skills/implement/SKILL.md` | 实现边界明确的代码变更 |
| `[F]` | `fix` | `.omp/skills/fix/SKILL.md` | 复现、定位并修复已知缺陷 |
| `[R]` | `review` | `.omp/skills/review/SKILL.md` | 审查 bug、风险、回归和验证缺口 |

## 5. Workflow 边界

当前 skills 的职责边界如下：

```text
discuss   -> 探索问题和方向
design    -> 基于已确认方向形成方案
implement -> 实现明确范围的代码变更
fix       -> 复现、定位并修复缺陷
review    -> 审查问题和风险，不默认修改
```

推荐流转：

```text
[D] discuss -> [P] design -> [I] implement -> [R] review
```

缺陷类任务可直接进入：

```text
[F] fix
```

用户可以随时使用新的 `[X]` 显式切换 skill，或使用 `[E]` 退出当前 skill。

## 6. 项目事实源

### 6.1 单项目事实源

每个项目的事实源为：

```text
projects/<project-id>/.project.yaml
```

最小字段：

```yaml
id: my_project
name: 我的项目
status: active
ownership: owned
```

推荐字段：

```yaml
description: null
tech_stack: null
main_entries: null
commands: null
relations: null
```

规则：

- `.project.yaml` 是人工事实源。
- AI 可以读取、解释和建议修改。
- AI 不应把讨论草稿、设计稿或索引内容反向当作事实源。

### 6.2 项目集索引

`registry.yaml` 是项目集索引，不是人工事实源。

它用于提供项目集层面的压缩视图。当前仓库中保留一个本地索引文件，包含已接入项目的核心字段。

原则：

- 当 `.project.yaml` 与 `registry.yaml` 不一致时，优先相信 `.project.yaml`。
- `registry.yaml` 可以由工具或人工重新生成。
- `registry.yaml` 不应反向覆盖项目事实源。

### 6.3 项目组

项目组入口为：

```text
programs/<program-id>/_program.yaml
```

项目组用于描述一组项目之间的组织关系。人工维护字段和生成字段应保持边界清晰。

## 7. 阅读资源与设计材料

`reading-resources/` 用于存放适合人类阅读的材料。

当前约定：

- `reading-resources/projects/`：按项目组织的阅读材料。
- `reading-resources/未分类/`：未归类材料、讨论草稿和设计草稿。

AI 写入规则：

- 只有任务或 active skill 明确需要时才写入。
- 讨论草稿和设计草稿记录结论，不记录完整聊天过程。
- 草稿不是事实源。
- 不应无任务需要地重写既有阅读材料。

`system-docs/` 只用于记录本系统自身的正式设计和稳定决策。

## 8. 主 Agent 与子 Agent

OMP 支持通过 task/subagent 机制进行主 agent 与子 agent 协作。

本系统建议的协作模型：

- 主 agent 负责理解用户目标、拆分任务、设置边界和汇总结果。
- 子 agent 负责独立调查、审查、验证或局部分析。
- 子 agent 默认只读。
- 只有主 agent 明确分配写入范围时，子 agent 才能修改文件。
- 多个子 agent 不应写入同一文件。
- 最终对用户负责的是主 agent。

适合使用子 agent 的场景：

- 大范围代码审查。
- 多模块调查。
- OMP 能力和项目现状对照。
- 独立验证或复核。

不适合使用子 agent 的场景：

- 单文件小改动。
- 快速问答。
- 需要严格串行写入的任务。

## 9. Git 与本地数据策略

当前 `.gitignore` 表达的策略：

- `registry.yaml` 默认本地生成，不提交。
- `projects/*` 默认本地保留。
- `programs/*` 默认本地保留。
- `reading-resources/*` 默认本地保留，但保留目录结构。
- `node_modules/`、`.vscode/` 和备份文件不提交。

该策略符合系统定位：本仓库主要管理个人本地项目集和 AI 协作协议，项目事实和阅读材料默认不作为公共仓库内容。

## 10. 当前文档入口

当前系统入口以实际文件为准：

- `AGENTS.md`：AI 常驻路由协议。
- `.omp/skills/*/SKILL.md`：按需加载的 OMP workflow。
- `projects/*/.project.yaml`：项目事实源。
- `programs/*/_program.yaml`：项目组入口。
- `registry.yaml`：项目集索引。
- `system-docs/项目集个人管理系统设计文档.md`：当前系统正式设计说明。

如果本设计文档与实际入口不一致，应优先检查实际入口文件，并同步更新本文档。
