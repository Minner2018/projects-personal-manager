# 项目集个人管理系统设计文档

## 1. 设计目标

`projects-personal-manager`，中文称为“项目集个人管理系统”，用于供个人在本地集中管理代码项目集。

系统的核心目标是：

- 索引项目。
- 理解项目。
- 组织 AI 上下文。
- 提供 AI 协作协议。

系统不承担以下职责：

- 不统一构建、测试、发布、依赖管理。
- 不接管项目自身 Git 仓库。
- 不规定项目内部目录结构。
- 不管理非代码项目。

项目源码可以集中放置在 `projects/` 下，但每个项目仍保持自治。

项目接入系统的最低要求是：

```text
projects/<project-id>/.project.yaml
```

近乎空项目也可以接入，只要提供合法的 `.project.yaml`。

## 2. 命名原则

目录和文档命名采用“中文优先、必要英文保留”的策略。

保留英文的情况：

- `README.md`。
- example 相关文件和路径，例如 `example_project`。
- 涉及脚本读取、生成或校验的字段、枚举值、脚本文件名。
- 简单常见或系统约定目录名，例如 `ai`、`human`、`projects`、`programs`、`workflows`、`tasks`、`memory`、`scripts`、`docs`。

使用中文的情况：

- 非 README、非 example、非约定入口、非项目 id 派生文件名的 `.md` 文档，默认使用中文。
- 协作语义较强或不够直观的子目录和文档可使用中文，例如 `角色入口/`、`未分类/`。

## 4. 项目事实源：`.project.yaml`

`.project.yaml` 是单项目人工事实源。

脚本和 AI 可以读取它、校验它、提出修改建议，但不应静默覆盖它。

`.project.yaml` 使用 YAML，字段命名统一使用 `snake_case`。自定义字段允许直接放在顶层，不强制放入 `custom`。

必填字段：

```yaml
id: example_project
name: 示例项目
status: active
ownership: owned
```

推荐字段：

```yaml
description: null
tech_stack: null
main_entries: null
commands: null
relations: null
```

推荐字段不强制填写，允许为 `null`。

### 4.1 枚举

```ts
type ProjectStatus =
  | "idea"
  | "active"
  | "paused"
  | "maintenance"
  | "archived";

type ProjectOwnership =
  | "owned"
  | "forked"
  | "external"
  | "reference";

type ProjectRelationType =
  | "depends_on"
  | "used_by"
  | "related_to"
  | "part_of"
  | "replaces";
```

### 4.2 项目关系

`relations` 每一项至少包含：

```yaml
target: other_project
type: depends_on
```

规则：

- `target` 可以填写目标项目的 `id` 或 `name`。
- 脚本解析时优先按 `id` 匹配。
- 按 `id` 找不到时，再按 `name` 匹配。
- 如果 `name` 匹配多个项目，脚本应生成警告，不自动选择。
- 生成 `registry.yaml` 时，`target` 应统一规范化为目标项目 `id`。
- `registry.yaml` 中仍使用字段名 `target`，不另起 `target_id`。

## 5. 项目集索引：`registry.yaml`

`registry.yaml` 是项目集唯一统一索引文件。

它不是事实源，而是由所有 `.project.yaml` 生成的本地索引。

规则：

- 位于项目集根目录。
- 默认不提交。
- 使用项目 `id` 作为 `projects` 下的 key。
- 只收集核心字段和推荐字段。
- 推荐字段缺失时，在 `registry.yaml` 中显式写为 `null`。
- 不保留 `.project.yaml` 中的自定义字段。
- 不记录 `generated_at`、`schema_version`、诊断信息或警告信息。
- `relations[].target` 统一写为项目 `id`。

可提交示例索引：

```text
registry.example.yaml
```

`registry.example.yaml` 只根据示例项目生成，用于展示索引结构。

## 6. 项目组：`programs/`

`programs/` 表示项目组。

每个项目组使用一个 `_program.yaml`：

```text
programs/<program-id>/_program.yaml
```

`_program.yaml` 是项目组主入口，同时包含人工维护字段和脚本维护字段。

人工维护字段：

```yaml
id: example_program
name: 示例项目组
description: 示例项目组用于演示 programs 目录的组织方式。
members:
  - example_project
```

规则：

- `members` 是项目组成员主字段。
- `members` 只允许填写项目 `id`。
- `id`、`name`、`description`、`members` 由人工维护。
- `projects` 由脚本维护。
- 脚本只允许更新 `projects` 字段。
- 脚本生成 `projects` 时优先读取根目录 `registry.yaml`。

脚本生成的 `projects` 每个项目只包含：

```yaml
id: example_project
name: 示例项目
status: active
ownership: owned
relations: null
```

`projects[].relations` 由脚本根据 `registry.yaml` 筛选生成，只保留 `target` 属于同一 `members` 的关系。指向项目组外部的关系不进入该项目组。

项目组关系类型复用 `ProjectRelationType`。

## 7. 人类材料区：`human/`

`human/` 存放适合人类阅读的文档、材料或资源。

结构以项目为单位，并保留未分类区域：

```text
human/
├── projects/
│   └── example_project/
│       └── 示例项目文档.md
└── 未分类/
```

规则：

- `human/projects/<project-id>/` 不要求入口文件。
- 项目目录内部不规定结构。
- `human/未分类/` 内部不规定结构。
- AI 可以按任务需要读取 `human/`。
- AI 只有在人类明确要求生成材料时，才向 `human/` 新增文件。
- AI 不应无任务需要地随意改写 `human/` 既有内容。

`human/projects/example_project/示例项目文档.md` 用于说明示例项目仅做参考使用。

## 8. AI 协作区：`ai/`

`ai/` 主要供 AI 明确职责、读取上下文、维护任务交接和记录长期记忆。

`ai/` 不是 AI 独占区。人类可以阅读、修改和整理其中内容。

最小结构：

```text
ai/
├── README.md
├── global.md
├── 角色入口/
├── workflows/
├── projects/
├── tasks/
└── memory/
```

### 8.1 `README.md` 与 `global.md`

`ai/README.md` 是 `ai/` 目录说明和导航。

`ai/global.md` 是 AI 执行任务时优先读取的全局上下文和工作协议。

### 8.2 角色入口

角色入口文档使用中文命名。

初始角色入口：

```text
ai/角色入口/
├── 设计.md
├── 实现.md
├── 测试.md
├── 审查.md
└── 维护.md
```

含义：

- `设计.md`：讨论、设计、需求澄清、方案比较。
- `实现.md`：代码修改、bug 修复、局部重构。
- `测试.md`：测试补充、测试失败分析、边界检查。
- `审查.md`：代码审查、架构一致性、安全和文档一致性检查。
- `维护.md`：维护 registry、项目组、AI 摘要、任务、记忆建议。

### 8.3 任务流程

`workflows/` 存放可复用任务流程。

角色入口与工作流程是松散的一对多关系：一个角色入口可以列出常用工作流程，但工作流程不在目录结构上绑定到某个角色。

初始工作流程：

```text
ai/workflows/
├── 系统-生成注册表.md
├── 系统-同步项目组.md
├── 系统-同步项目摘要.md
├── 系统-检查一致性.md
├── 系统-任务交接.md
├── 功能实现.md
├── 缺陷修复.md
├── 补充测试.md
└── 代码审查.md
```

系统维护类工作流程使用 `系统-` 前缀，用于区分项目集系统维护流程和普通代码工作流程。

### 8.4 上下文预算

系统保留四级上下文预算，但不创建独立的上下文包目录。规则写入 `ai/global.md`、角色入口和工作流程。

上下文预算级别：

```text
极简
项目
项目组
深度
```

含义：

- `极简`：新会话、简单问答、项目定位。读取 `ai/README.md`、`ai/global.md`、`registry.yaml`。
- `项目`：单项目开发、分析、测试和问答。在极简基础上读取 `ai/projects/<project-id>.md`、`projects/<project-id>/.project.yaml`。
- `项目组`：项目组级分析和多项目协作。在项目基础上读取 `programs/<program-id>/_program.yaml` 和相关成员摘要。
- `深度`：复杂 bug、重构、迁移、性能、安全审查等。必要时读取关键源码、测试、配置、`human/` 材料和相关任务记录。

## 9. AI 项目摘要

AI 项目摘要位于：

```text
ai/projects/<project-id>.md
```

它由脚本生成骨架，可由人类或 AI 补充内容。

它不是事实源，而是面向 AI 的项目压缩说明。

文件关系：

- `.project.yaml` 是每个项目的身份证。
- `_program.yaml` 是项目组的户口本。
- `registry.yaml` 是项目集的全民记录。
- `ai/projects/<project-id>.md` 是项目的个人简历。

AI 阅读项目时应优先参考 `.project.yaml`。如果发现 `.project.yaml`、`registry.yaml`、AI 项目摘要之间存在不一致，应告知人类，而不是静默修正事实源。

项目摘要可包含近期任务区域，用于导航到 `ai/tasks/`，但不复制任务正文。

建议模板：

```markdown
# 项目 AI 摘要：<project-id>

## 元信息
- 摘要状态：draft / confirmed / stale
- 最后核验：YYYY-MM-DD
- 已检查来源：.project.yaml, registry.yaml

## 摘要

## 事实
- ID:
- 名称:
- 项目状态:
- 归属:
- 说明:

## 技术栈

## 入口

## 命令

## 关系

## 近期任务

## 关键说明

## AI 阅读指引

## 约束
```

摘要状态使用独立枚举：

```ts
type AiSummaryStatus =
  | "draft"
  | "confirmed"
  | "stale";
```

## 10. AI 任务与记忆

### 10.1 任务记录

`ai/tasks/` 存放任务记录和任务交接文件。初始阶段保持平目录，不拆分状态子目录或模板子目录。

任务文件初始模板：

```markdown
# 任务：<task-name>

## 元信息
- 项目：<project-id>
- 角色入口：设计 / 实现 / 测试 / 审查 / 维护
- 工作流程：<工作流程名称>
- 任务状态：进行中 / 已完成 / 受阻
- 创建日期：YYYY-MM-DD
- 更新日期：YYYY-MM-DD

## 目标

## 上下文

## 相关文件

## 决策

## 变更

## 测试

## 风险

## 交接
```

### 10.2 长期记忆

`ai/memory/` 存放长期记忆，分为全局记忆和分项目记忆：

```text
ai/memory/
├── global.md
├── global.example.md
└── projects/
    └── example_project.md
```

规则：

- `ai/memory/global.md` 是真实全局记忆，默认不提交。
- `ai/memory/global.example.md` 是可提交示例。
- `ai/memory/projects/example_project.md` 是示例项目记忆。
- 长期记忆只记录跨会话可复用、长期有效、对后续工作有明确价值的内容。
- 长期记忆不替代任务记录，也不保存临时推理过程。

所有 `.example.md` 文件都应在文件内说明：该文件是示例文件，不代表真实项目事实、真实任务记录或真实长期记忆。

## 11. 自动化脚本

项目使用 Bun 运行 TypeScript 脚本。脚本文件使用 `.ts`，由 Bun 直接执行。

`tsconfig.json` 用于 VS Code 和 `bun run typecheck` 的类型检查。当前依赖：

- `typescript`
- `@types/node`

核心脚本：

```text
scripts/
├── _shared.ts
├── generate-registry.ts
├── generate-registry-example.ts
├── sync-programs.ts
├── sync-ai-projects.ts
└── check-consistency.ts
```

脚本命令：

```text
bun run registry
bun run registry:example
bun run programs
bun run ai-projects
bun run check
bun run typecheck
```

`_shared.ts` 存放 YAML 读写、registry 构建、备份写入、关系 target 规范化和摘要补全等共享逻辑。

`generate-registry-example.ts` 只根据示例项目生成 `registry.example.yaml`，脚本自身按当前决定忽略，不作为仓库提交内容。

职责：

- `_shared.ts`：提供脚本共享工具，不直接作为命令运行。
- `generate-registry.ts`：从 `projects/*/.project.yaml` 生成根目录 `registry.yaml`。
- `generate-registry-example.ts`：只根据示例项目生成根目录 `registry.example.yaml`。
- `sync-programs.ts`：根据 `registry.yaml` 更新 `programs/*/_program.yaml` 中由脚本维护的 `projects` 字段。
- `sync-ai-projects.ts`：根据 `.project.yaml` 或 `registry.yaml` 生成或补全 `ai/projects/<project-id>.md` 骨架。
- `check-consistency.ts`：检查 `.project.yaml`、`registry.yaml`、`_program.yaml`、AI 项目摘要之间的不一致。

脚本写入原则：

- `generate-registry.ts` 可以覆盖 `registry.yaml`。
- `generate-registry-example.ts` 可以覆盖 `registry.example.yaml`。
- `sync-programs.ts` 只允许覆盖 `_program.yaml` 中的 `projects` 字段。
- `sync-ai-projects.ts` 可以创建新摘要文件，或为已有摘要补充缺失骨架章节，但不覆盖已有人工内容。
- `check-consistency.ts` 只报告问题，不修改文件。
- 脚本修改既有文件前应先备份原文件。
- 出错则中止、输出错误信息、保留备份。
- 成功则删除备份。
- 对人工维护区域，脚本只能提示或补缺，不应直接覆盖。

## 12. 一致性检查

第一版检查范围如下。

### 12.1 `.project.yaml` 自身合法性

- 是否存在。
- YAML 是否可解析。
- 是否包含必填字段：`id`、`name`、`status`、`ownership`。
- 字段名是否为 `snake_case`。
- `status` 是否在枚举中。
- `ownership` 是否在枚举中。
- `relations[].type` 是否在枚举中。
- `relations[].target` 是否能按 `id` 或 `name` 匹配到项目。

### 12.2 `registry.yaml` 与 `.project.yaml` 是否一致

- `registry.yaml` 是否存在。
- registry 中是否包含所有项目。
- registry 中是否包含已经不存在的项目。
- 核心字段和推荐字段是否与 `.project.yaml` 一致。
- 推荐字段缺失时是否显式为 `null`。
- relations target 是否已规范化为项目 `id`。
- 是否保留了自定义字段；如保留则报错。

### 12.3 `_program.yaml` 与 `registry.yaml` 是否一致

- `members` 中的项目 id 是否存在于 `registry.yaml`。
- `projects` 是否只包含 `id`、`name`、`status`、`ownership`、`relations`。
- `projects` 是否由 `members` 完整生成。
- `projects[].relations` 是否只包含 target 属于同一项目组的关系。
- 是否有组外关系混入。

### 12.4 AI 项目摘要与事实源是否明显冲突

- 对应项目是否存在。
- 元信息是否存在。
- 事实区中的 ID、名称、项目状态、归属如能识别，应与 `.project.yaml` 一致。
- 如果不一致，只报告，不自动修改。
- 如果摘要状态为 `confirmed`，但事实不一致，应提示摘要可能过期。

### 12.5 example 规则

- `projects/example_project/.project.yaml` 是否存在。
- `projects/example_project/index.ts` 是否存在。
- `registry.example.yaml` 是否只包含 example 项目。
- `programs/example_program/_program.yaml` 是否存在。
- example 任务、记忆、human 项目示例、human 未分类示例是否存在。
- `generate-registry-example.ts` 是否未被提交；该项只有在 Git 仓库中才可检查。

检查结果分为：

- `error`：必须修复，否则生成物或事实关系不可靠。
- `warning`：建议修复，但不一定阻止继续工作。

## 13. Git 跟踪策略

真实项目事实、真实源码、真实 AI 记忆和真实任务记录默认不提交。example 相关内容可以提交。

当前 `.gitignore` 核心规则建议：

```gitignore
# 本地生成的项目集索引
/registry.yaml

# 依赖目录
/node_modules/

# 真实项目默认只保留在本地
/projects/*
!/projects/example_project/
```

### 16.1 根目录 `README.md`

````markdown
# projects-personal-manager

项目集个人管理系统用于供个人在本地集中管理代码项目集。

系统只负责项目索引、项目理解、AI 上下文组织和 AI 协作协议，不接管各项目自身的构建、测试、发布、依赖管理或 Git 仓库。

## 核心目录

- `programs/`：项目组目录。每个项目组使用 `_program.yaml` 描述成员和组内项目关系。
- `human/`：适合人类阅读的材料、文档或资源。
- `ai/`：供 AI 读取的全局上下文、角色入口、工作流程、项目摘要、任务记录和长期记忆。
- `scripts/`：维护索引、项目组、AI 摘要和一致性检查的脚本。

## 接入项目

在 `projects/<project-id>/` 下创建 `.project.yaml`。

最小示例：

```yaml
id: example_project
name: 示例项目
status: active
ownership: owned
```

推荐字段：

```yaml
description: null
tech_stack: null
main_entries: null
commands: null
relations: null
```

- `bun run programs`：同步项目组 `_program.yaml` 中的 `projects` 字段。
- `bun run ai-projects`：生成或补全 AI 项目摘要。
- `bun run check`：检查核心文件之间的一致性。
- `bun run typecheck`：检查脚本类型。
````

### 16.2 `ai/README.md`

```markdown
# AI 协作区

本目录用于帮助 AI 明确工作职责、选择上下文、执行工作流程、记录任务交接和读取长期记忆。

本目录不是 AI 独占区。人类可以阅读、修改和整理其中内容。

## 目录

- `global.md`：AI 执行任务时优先读取的全局上下文和工作协议。
- `角色入口/`：按角色组织的任务入口。
- `workflows/`：可复用工作流程。
- `projects/`：面向 AI 的项目摘要。
- `tasks/`：任务记录和交接文件。
- `memory/`：长期记忆。

## 使用原则

AI 开始任务时，应先确认任务类型，再选择合适的角色入口和上下文预算。

AI 读取项目时，应优先参考 `projects/<project-id>/.project.yaml`。如果发现 `.project.yaml`、`registry.yaml`、`_program.yaml` 或 AI 项目摘要之间存在不一致，应告知人类，不应静默修正事实源。

AI 只有在人类明确要求生成材料时，才向 `human/` 新增文件。
```

### 16.3 `ai/global.md`

```markdown
# AI 全局上下文

## 系统定位

这是一个供个人在本地集中管理代码项目集的系统，只负责项目索引、项目理解、AI 上下文组织和 AI 协作协议。

系统不统一构建、测试、发布、依赖管理，不接管项目 Git，也不规定项目内部结构。

## 事实源优先级

- `.project.yaml` 是单项目事实源。
- `registry.yaml` 是由 `.project.yaml` 生成的项目集索引。
- `_program.yaml` 是项目组入口，其中 `members` 由人工维护，`projects` 由脚本维护。
- `ai/projects/<project-id>.md` 是面向 AI 的项目摘要。

除 `.project.yaml` 的人工内容外，其余文件都不能反向覆盖事实源。发现不一致时，应告知人类。

## 上下文预算

- `极简`：读取 `ai/README.md`、`ai/global.md`、`registry.yaml`。
- `项目`：在极简基础上读取 `ai/projects/<project-id>.md`、`projects/<project-id>/.project.yaml`。
- `项目组`：在项目基础上读取 `programs/<program-id>/_program.yaml` 和相关成员摘要。
- `深度`：必要时读取关键源码、测试、配置、`human/` 材料和相关任务记录。

## 工作原则

- 不假设，不掩盖困惑；明确说明判断、缺口和取舍。
- 用最少代码解决问题，不做推测性开发。
- 只改必要文件，只清理自己造成的问题。
- 先定义成功标准，再迭代到验证通过；无法验证时说明原因。
- 不静默覆盖人工维护内容；生成文件与人工文件冲突时先报告。
- 长期记忆只记录长期有效、跨会话可复用的信息。

## 写入边界

- 可按任务读取 `human/`；只有在人类要求生成材料时才写入。
- 可创建或补全 AI 项目摘要骨架，但不覆盖已有人工内容。
- 可记录任务交接；不把临时推理写入长期记忆。
- 可建议更新长期记忆；真实长期记忆默认不提交。
```
