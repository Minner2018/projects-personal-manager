# AI 全局上下文

## 系统定位

本系统用于本地管理代码项目集，只负责项目索引、项目理解、AI 上下文组织和 AI 协作协议。

系统不统一构建、测试、发布、依赖管理，不接管项目 Git，不规定项目内部结构。

## `ai/` 目录

- `global.md`：AI 执行任务时优先读取的全局上下文和工作协议。
- `角色入口/`：按角色组织的任务入口。
- `workflows/`：设计、实现、测试、维护等可复用工作流程。
- `projects/`：面向 AI 的项目摘要。
- `tasks/`：任务记录和交接文件。
- `locks/`：AI 并发工作时使用的文件级锁声明。
- `memory/`：长期记忆。

## 事实源优先级

- `.project.yaml` 是单项目事实源。
- `registry.yaml` 是由 `.project.yaml` 生成的项目集索引。
- `_program.yaml` 是项目组入口，其中 `members` 由人工维护，`projects` 由脚本维护。
- `ai/projects/<project-id>.md` 是面向 AI 的项目摘要。

生成文件不能反向覆盖事实源。发现不一致时，应告知人类。

## 上下文预算

- `极简`：读取 `ai/global.md`、`registry.yaml`。
- `项目`：在极简基础上读取 `ai/projects/<project-id>.md`、`projects/<project-id>/.project.yaml`。
- `项目组`：在项目基础上读取 `programs/<program-id>/_program.yaml` 和相关成员摘要。
- `深度`：必要时读取项目记忆、关键源码、测试、配置、`human/` 材料和相关任务记录。

## 工作原则

- 明确判断、缺口和取舍，不掩盖不确定性。
- 用最少改动解决问题，不做推测性开发。
- 只改必要文件，只清理自己造成的问题。
- 先定义成功标准，再验证；无法验证时说明原因。
- 不静默覆盖人工维护内容。
- 并行工作时先声明任务和计划修改文件；目标文件被声明或锁定时先报告冲突。
- 长期记忆只记录长期有效、跨会话可复用的信息。

## AI 文件锁

`ai/locks/` 用于多个 AI 并行工作时声明文件级占用。锁文件只表示当前占用或需要确认，不是事实源，不得覆盖项目事实、索引或源码。

并发协作规则：

1. 修改文件前运行 `bun run check:concurrency`。
2. 在任务记录中声明 `## 计划修改文件`。
3. 高冲突风险文件优先用脚本创建锁：

```powershell
bun run lock:create -- --file <path> --task <task> --owner <owner> --reason <reason>
```

4. 发现目标文件被其他任务声明或锁定时，停止修改并报告冲突。
5. 完成、暂停或放弃相关修改时，释放自己创建的锁。

锁文件字段为 `owner`、`task`、`file`、`created_at`、`expires_at`、`reason`。`--hours` 可设置过期小时数，默认 `2`。锁过期只代表需要人工确认，不代表可以自动覆盖。

## 写入边界

- `human/`：可按任务读取；只有在人类要求生成材料时才写入。
- `system-docs/`：可记录该系统本身的正式设计。
- `ai/projects/`：可创建或补全摘要骨架，不覆盖已有人工内容。
- `ai/tasks/`：可记录任务交接。
- `ai/locks/`：可创建、更新和释放自己的锁；不删除其他执行者的锁。
- `ai/memory/`：只建议记录长期有效信息，真实长期记忆默认不提交。
